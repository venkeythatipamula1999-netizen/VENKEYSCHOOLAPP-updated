Dependency Vulnerabilities (npm audit)

HIGH FIXED xlsx — Prototype Pollution & ReDoS
The xlsx package had two critical flaws: Prototype Pollution (GHSA-4r6h-8v6p-xvw6) and Regular Expression Denial of Service (GHSA-5pgg-2g8v-p4x9). No patched version of xlsx was published by the maintainer.
Fix: Removed xlsx entirely. Replaced with exceljs (actively maintained, no known vulnerabilities) in the student bulk-upload endpoint. All xlsx parsing logic was rewritten using ExcelJS's async streaming API which also prevents formula injection attacks.
LOW FIXED tar@6.2.1 — Multiple CVEs
The tar package at version 6.2.1 was vulnerable to multiple security issues. It was pulled in transitively by expo via @expo/cli → cacache → tar.
Fix: Added an overrides block in package.json forcing all instances of tar to resolve to ^7.5.11. Confirmed via npm ls tar — no instance of 6.x remains.
LOW ×10 ACCEPTED RISK @tootallnate/once — Incorrect Control Flow Scoping
Ten low-severity advisories all trace to @tootallnate/once <3.0.1 buried deep in the firebase-admin dependency tree: firebase-admin → @google-cloud/firestore → google-gax → @google-cloud/storage → teeny-request → http-proxy-agent → @tootallnate/once. Advisory: GHSA-vpq2-c234-7xj6
Accepted Risk: The only npm-available fix is to downgrade firebase-admin to v10.3.0 — a major breaking change incompatible with the current codebase. The vulnerability is also low severity and only affects control-flow in edge cases within the proxy agent, not exploitable via normal school app traffic flows. Recommend monitoring for a firebase-admin v14+ release that resolves this chain.

Code-Level Security Fixes (Full Navigation Flow Scan)

MEDIUM FIXED IDOR — /api/complete-profile allowed any user to update any profile
Any authenticated user could pass an arbitrary docId to overwrite another user's profile data (name, mobile, blood group, emergency contact, DOB).
Fix: Added ownership check — the JWT user's ID or UID must match the docId/uid being updated. Any mismatch returns HTTP 403.
HIGH FIXED Missing Role Check — DELETE /api/classes/:id
Any authenticated user (teacher, driver, parent) could delete any class record by sending a DELETE request with any class document ID.
Fix: Added principal / admin role guard. Returns HTTP 403 for all other roles.
HIGH FIXED Missing Role Check — POST /api/students (add student)
Any authenticated user could create new student records in the database.
Fix: Restricted to principal / admin only.
HIGH FIXED Missing Role Check — DELETE /api/students/:id
Any authenticated user could permanently delete any student record.
Fix: Restricted to principal / admin only.
MEDIUM FIXED Missing Role Check — POST /api/students/bulk-upload/:classId
Any authenticated user could bulk-upload student data (CSV/XLSX) to any class.
Fix: Restricted to principal / admin only.
HIGH FIXED Missing Role Check — POST /api/assign-classes (teacher class assignment)
Any authenticated user could reassign any teacher's class assignments — including removing all classes from another teacher.
Fix: Restricted to principal / admin only.
MEDIUM FIXED Missing Role Check — POST /api/set-class-teacher
Any authenticated user could assign or remove class-teacher status for any teacher.
Fix: Restricted to principal / admin only.
MEDIUM FIXED Missing Role Check — POST /api/save-timetable
Any authenticated user could overwrite any teacher's timetable, causing widespread calendar corruption and notification spam.
Fix: Restricted to principal / admin only.
MEDIUM FIXED Missing Role Check — POST /api/onboard-teacher
Any authenticated user could create new teacher / staff accounts and generate valid login IDs (TCH-XXXX), bypassing the principal's gatekeeping.
Fix: Restricted to principal / admin only.
MEDIUM FIXED Missing Role Check — GET /api/onboarded-users (PII exposure)
Any authenticated user (e.g. a parent or driver) could retrieve the full staff directory including names, emails, phone numbers, blood groups, and emergency contacts.
Fix: Restricted to principal / admin only.
MEDIUM FIXED Missing Role Check — POST /api/delete-user & /api/add-logistics-staff
Any authenticated user could mark any staff member inactive (/api/delete-user) or add new logistics staff (/api/add-logistics-staff).
Fix: Both endpoints restricted to principal / admin only.
LOW FIXED Excessive JWT Token Lifespan (30 days → 8 hours)
JWT tokens had a 30-day expiry. A stolen token would remain valid for a month, giving an attacker a large window of access to all user data and admin operations.
Fix: Reduced expiresIn from '30d' to '8h'. Limits breach exposure window to a single work-day session.

Recommendations — Future Hardening

RECUpgrade firebase-admin when v14+ available
Monitor for a firebase-admin release that updates its google-gax and teeny-request chain to fix the @tootallnate/once vulnerability. This will clear the remaining 10 low-severity advisories.
RECImplement refresh tokens
JWT tokens are now 8h. Consider adding a refresh-token mechanism so users don't have to re-login mid-day, while keeping the security window tight.
RECAdd request logging / SIEM integration
Currently errors are logged to console. Consider integrating a structured logging solution (e.g. Winston → Cloud Logging) and alerting on repeated 403/401 patterns which indicate probing.
RECEnforce HTTPS-only cookies for tokens
If authentication tokens are ever stored in cookies (currently in Authorization headers), ensure HttpOnly; Secure; SameSite=Strict flags are set.
